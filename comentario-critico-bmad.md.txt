“Mi hipótesis inicial era…”

Mi hipótesis inicial era que BMAD funcionaría como un generador automático de código que aceleraría directamente la implementación del feature, sin necesidad de tanta planificación previa ni documentación estructurada.

“Lo que ocurrió realmente fue…”

Lo que ocurrió realmente fue que BMAD siguió un flujo estructurado tipo Agile: primero detectó que no había PRD, luego creó un PRD (prd.md), generó épicas (epics.md), creó historias de desarrollo y finalmente implementó el código con tests. En lugar de ir directo al código, el agente priorizó la documentación, validación y readiness antes de la implementación.

“La mayor fortaleza observada fue…” (con evidencia)

La mayor fortaleza observada fue la trazabilidad y organización del desarrollo.
Evidencia: el sistema creó automáticamente archivos como:

prd.md (documentación del producto)

epics.md

historia de desarrollo (1-1-guided-prd-creation.md)

código + tests
Esto demuestra que BMAD mantiene coherencia entre requisitos, planificación y desarrollo, reduciendo la improvisación típica de prompts sueltos.

“La mayor debilidad observada fue…” (con evidencia)

La mayor debilidad observada fue el exceso de overhead documental para un feature relativamente simple.
Evidencia: el agente no podía validar nada porque “no había PRD”, lo que obligó a crear primero documentación antes de implementar, alargando el proceso respecto a un flujo tradicional con Copilot directo al código.

“El punto de fricción principal fue…”

El punto de fricción principal fue la especificación (spec) y el flujo PRD-first.
El agente necesitaba continuamente confirmar:

qué feature usar

idioma del PRD

si generar épicas desde plantilla o PRD
Esto ralentizó el proceso inicial.

“El tipo de error más frecuente fue…”

El tipo de error más frecuente fue la falta de contexto del feature (ambigüedad de requisitos), lo que provocó que el agente usara el feature por defecto “Guided PRD Creation” en lugar de uno definido explícitamente por el usuario.

“Qué tuve que hacer yo (humano) para que funcionara fue…”

Tuve que:

confirmar el feature a documentar

elegir el idioma del PRD (español)

aprobar edición de archivos sensibles (package.json)

autorizar ejecución de tests y dependencias
Es decir, el humano actuó como Product Owner validando decisiones del agente.

“En qué contexto SÍ la usaría…”

Sí usaría BMAD en:

proyectos medianos o grandes

desarrollo con equipo

proyectos que requieren documentación formal (PRD, épicas, historias)

TFG, proyectos académicos o empresariales con trazabilidad

“En qué contexto NO la usaría…”

No la usaría en:

scripts pequeños

ejercicios rápidos de clase

prototipos rápidos (MVP en pocas horas)

features muy simples donde el PRD es innecesario

“Qué cambiaría del workflow/prompts/reglas para mejorar…”

Cambiaría:

usar Quick Flow por defecto en features pequeños

pedir el feature desde el inicio para evitar PRD genérico

reducir la cantidad de confirmaciones intermedias

integrar mejor con Copilot/VSCode sin depender tanto de plantillas PRD
